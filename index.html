Building My First Pentesting Lab — Windows + Linux Terminal

  • Built the base lab-htb/ structure (recon, loot, notas) on both Windows and Linux, entirely from the command line — no mouse
  • Practiced navigation and file ops on both systems: cd, pwd, dir/ls, mkdir, del/rm, rmdir, rm -r, Get-Help
  • Goal: a repeatable methodology for evidence and findings, ready before touching an actual HTB/TryHackMe box
  • The error that taught the most: mkdir [-loot][-notas] in PowerShell didn't create two folders — square brackets are wildcard metacharacters, not literal text, so it created one folder literally named [-loot][-notas] — a "ghost folder" that del [-loot][-notas] ran against without error yet never deleted, because the pattern matched nothing
  • Fix and new fixed rule: preview with a real wildcard before deleting — dir *notas* to see what matches, only then del *notas*
  • Same lesson from the Linux side: Bash does brace expansion ({}) for multiple names, PowerShell/CMD don't; Linux is case-sensitive (config ≠ Config) where Windows is more permissive
Key insight: The terminal doesn't interpret what I meant to type — it executes exactly what I typed. "See before you destroy": confirm with dir/ls what a wildcard actually matches before running the delete. Many bugs — and many vulnerabilities — start from assuming two environments behave the same when they don't.
Concepts

Hypervisors: The Invisible Building Owner

  • Reframed the question from "what is a hypervisor" to "how does one machine convince several operating systems each has the whole hardware to itself" — that's where the building analogy clicked: hardware = the building, each VM = a floor, each OS = a tenant, and the hypervisor = the owner, who doesn't live on any floor but sits above all of them, deciding how much of the building each tenant gets and keeping floors from interfering with each other
  • Type 1 (bare-metal) hypervisors run directly on the hardware — the real owner, answering to no one; Type 2 hypervisors run on top of a host OS, like VirtualBox — more like a hired manager who still answers to the actual owner
  • The puzzle that didn't add up: the kernel runs at Ring 0, the highest privilege level — so how can several kernels each believe they're the one in charge at the same time?
  • The answer: Ring -1, a privilege level above Ring 0, is where the hypervisor actually runs — handing out CPU time, memory and physical devices while every VM still believes it's alone on the hardware
  • Personal click: Kali on WSL2 isn't just "another Linux window" — it literally runs on Hyper-V, Windows' own hypervisor, used for months without registering what was actually running underneath
Key insight: Virtualisation and an OS isolating processes from each other are the same principle — isolation — applied at different layers: each layer hides the complexity below it to offer a controlled, convincing environment above. Understanding the problem a technology solves is what turns a definition from something to memorize into something that actually makes sense.
