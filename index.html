Conceptos

Montando mi primer laboratorio de pentesting — Terminal Windows + Linux

  • Construí la estructura base de lab-htb/ (recon, loot, notas) tanto en Windows como en Linux, todo desde la línea de comandos — sin tocar el mouse
  • Practiqué navegación y operaciones de archivos en ambos sistemas: cd, pwd, dir/ls, mkdir, del/rm, rmdir, rm -r, Get-Help
  • Meta: una metodología repetible para evidencia y hallazgos, lista antes de tocar una máquina real de HTB/TryHackMe
  • El error que más enseñó: mkdir [-loot][-notas] en PowerShell no creó dos carpetas — los corchetes son metacaracteres comodín, no texto literal, así que creó una sola carpeta llamada literalmente [-loot][-notas] — una "carpeta fantasma" contra la que del [-loot][-notas] corría sin error pero nunca la borraba, porque el patrón no coincidía con nada
  • Solución y nueva regla fija: previsualizar con un comodín real antes de borrar — dir *notas* para ver qué coincide, y solo entonces del *notas*
  • La misma lección desde el lado de Linux: Bash sí hace expansión de llaves ({}) para generar varios nombres, PowerShell/CMD no; Linux distingue mayúsculas y minúsculas (config ≠ Config) donde Windows es más permisivo
Key insight: La terminal no interpreta lo que quise escribir — ejecuta exactamente lo que escribí. "Ver antes de destruir": confirmar con dir/ls qué coincide realmente con un comodín antes de correr el borrado. Muchos bugs — y muchas vulnerabilidades — nacen de asumir que dos entornos se comportan igual cuando no es así.
Conceptos

Hipervisores: el dueño invisible del edificio

  • Cambié la pregunta de "¿qué es un hypervisor?" a "¿cómo puede una sola computadora convencer a varios sistemas operativos de que cada uno tiene el hardware completo para sí mismo?" — ahí encajó la analogía del edificio: el hardware es el edificio, cada VM es un piso, cada sistema operativo es un inquilino, y el hypervisor es el dueño, que no vive en ningún piso sino que está por encima de todos, decidiendo cuánto espacio recibe cada inquilino y evitando que un piso interfiera con otro
  • Los hypervisors Tipo 1 (bare-metal) corren directamente sobre el hardware — el verdadero dueño, que no le rinde cuentas a nadie; los Tipo 2 corren encima de un sistema operativo anfitrión, como VirtualBox — más parecido a un administrador contratado que sigue dependiendo del dueño real
  • El rompecabezas que no cuadraba: el kernel corre en Ring 0, el nivel de privilegios más alto — entonces, ¿cómo pueden varios kernels creer al mismo tiempo que cada uno es el que manda?
  • La respuesta: existe el Ring -1, un nivel de privilegios por encima de Ring 0, y ahí es donde realmente corre el hypervisor — repartiendo tiempo de CPU, memoria y dispositivos físicos mientras cada VM sigue creyendo que está sola en el hardware
  • Conexión personal: mi Kali en WSL2 no es solo "otra ventana con Linux" — literalmente corre sobre Hyper-V, el hypervisor de Windows, usado durante meses sin darme cuenta de qué corría realmente por debajo
Key insight: La virtualización y un sistema operativo aislando procesos entre sí son el mismo principio — isolation — aplicado en capas distintas: cada capa oculta la complejidad de abajo para ofrecer un entorno controlado y convincente arriba. Entender el problema que resuelve una tecnología es lo que convierte una definición de algo para memorizar en algo que realmente tiene sentido.
